## 背景

- 目标是把“实时语音”做成一个可复用的网关层：把上游音频输入统一成稳定的流式协议，再把下游 ASR/TTS/LLM 等能力以可控的并发、超时与回压接起来
- 语音链路的工程难点不在于“能不能跑”，而在于低延迟 + 稳定性：抖动、丢包、慢消费者、突发并发都会把尾延迟放大
- 选择 Rust 的原因是把关键路径做成可控的系统工程：内存生命周期、并发模型、背压策略、以及观测都更容易落地

## 架构设计

- 连接层
  - 负责会话生命周期、鉴权、限流、以及连接级别的超时与熔断
  - 将上游输入统一成“固定大小的音频帧/块”，方便下游按节拍消费
- 流水线层
  - 以“可中断的流式任务”组织：输入 → 预处理 → 推理 → 输出
  - 每一段都必须有回压：队列长度上限 + 丢弃/降级策略，避免无限堆积
- 资源与并发
  - 把 CPU 密集与 I/O 密集拆开：避免把 reactor 卡死
  - 用明确的并发上限保护下游推理服务，保证 P99 不被拖垮
- 可观测性
  - 至少记录：入站速率、丢弃率、队列长度、端到端延迟分位数、下游错误码分布

## 踩坑记录

- “吞吐很高但体验很差”：平均延迟下降不代表可用，语音更在乎 P95/P99 和抖动（jitter）
- “背压没做好导致雪崩”：慢消费者会把队列撑爆，最终表现为延迟线性增长直至超时
- “日志把系统拖慢”：高频链路打日志会变成隐形瓶颈，必须抽样/聚合并把落盘从关键路径移走
- “下游不稳定引发连锁反应”：超时与重试要可控，否则会放大下游压力

## 最终成果

- 能稳定承载多会话并发的实时语音链路，并且对延迟与抖动有可观测的指标闭环
- 形成一套可复用的网关模板：连接管理、回压策略、并发控制、降级与观测
