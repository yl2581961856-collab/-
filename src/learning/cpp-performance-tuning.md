## 目标与边界

- 优先目标：把性能问题从“感觉慢”变成“可量化、可回归”的指标（吞吐/延迟/CPU%/内存/带宽）
- 边界条件：先不做算法级改动，先把工程侧的隐藏开销（拷贝、分配、锁、I/O）清干净
- 结果验收：每次优化都要有基准对比（同机器、同输入、同编译参数）

## 常见瓶颈

- 低效内存访问：cache miss、跨 NUMA、错误的结构体布局（AoS/SoA）
- 频繁分配与碎片：`new/delete`、小对象过多、`std::string` 临时对象
- 锁争用与伪共享：热点计数器、全局队列、false sharing
- 过多拷贝：不必要的值传递、容器扩容、序列化/反序列化
- 编译器没帮你优化：未开启 LTO/PGO、禁止了向量化、抽象层导致无法内联
- I/O 被忽略：同步落盘、日志过量、读取路径没有预取/批量

## 优化手段

- 测量优先
  - wall-time：基准测试（固定输入、固定线程数）
  - CPU profile：`perf` / `vtune` / `pprof`（先找热点再改）
  - cache/分支：`perf stat` 看 cache-misses、branch-misses
- 内存与布局
  - 小对象池/arena：把碎片与锁争用从关键路径挪走
  - SoA vs AoS：按访问模式组织数据，减少无效加载
  - 对齐与 padding：把热点字段放一起，避免伪共享
- 并发
  - 降低共享状态：能局部就局部，能分片就分片
  - 用无锁/低锁结构：但以可维护性为前提，避免过度复杂
- 编译与链接
  - `-O2/-O3`、`-march=native`、LTO、PGO（按项目成熟度逐步引入）
  - 控制模板与头文件膨胀：减少编译时间与 I-cache 压力
