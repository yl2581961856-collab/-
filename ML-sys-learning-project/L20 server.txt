参数解释（先理解边界，再开始调参）
--tensor-parallel-size 4：权重切分到 4 张卡，推理时跨卡同步
--gpu-memory-utilization 0.78：给 KV Cache / graph capture 预留空间；过高更容易 OOM/抖动
--shm-size：容器共享内存；多卡编译/初始化阶段不稳时要优先排查这一项（见 [S4]）
             --shm-size：过小会导致 NCCL 同步异常（后文真实踩坑）

[S3] LoRA/QLoRA 微调（Swift + DeepSpeed）
核心目标：在 L20×4 上把 “能跑 → 稳定 → 可回归” 做成闭环。先用短上下文与小数据跑通，再逐步抬 max_length/分辨率/并发。

建议每次实验固定记录 4 个东西（第三方视角最需要）
1) 训练命令（完整可粘贴）
2) 关键超参：max_length、有效 batch、lr、lora_rank/lora_alpha、是否 checkpointing、是否 ZeRO-3
3) 训练产物路径（checkpoint 目录）
4) 推理回归方式（固定 prompt/图片集 + JSONL 输出）

数据集 JSONL（多模态）写法参考（按报错对齐字段）
写法 A：messages + 多模态 content（更接近 OpenAI / Qwen-VL 的消息体）
```json
{"id":"ex-0001","messages":[
  {"role":"user","content":[
    {"type":"image","image":"/data/infer_images/0001.jpg"},
    {"type":"text","text":"请描述图片中车辆的颜色与大致场景。"}
  ]},
  {"role":"assistant","content":[
    {"type":"text","text":"车辆为白色，场景像是城市道路/停车区域。"}
  ]}
]}
```

写法 B：images + conversations（部分数据管线更偏这个结构）
```json
{"id":"ex-0001","images":["/data/infer_images/0001.jpg"],"conversations":[
  {"from":"user","value":"<image>\n请描述图片中车辆的颜色与大致场景。"},
  {"from":"assistant","value":"车辆为白色，场景像是城市道路/停车区域。"}
]}
```

回归验证（不合并 LoRA，直接当“补丁”验证）
1) 固定一组输入（图片目录 + prompt 模板 + 并发）输出 JSONL
2) 对比 base vs lora：关注错误类型是否减少、是否出现新的幻觉/不遵循格式
3) 如果要做快速 A/B：用同一套 infer_batch_strict.py 跑两次，只换 served-model-name 或 lora adapter

参数解释（按“最容易踩坑”的顺序）
--per_device_train_batch_size：最直接的激活显存开关，先从 1 起步
--gradient_accumulation_steps：用时间换显存；有效 batch = micro_batch × acc_steps × GPU 数
--torch_dtype bfloat16：bf16 更稳；显存占用与 fp16 同级
--gradient_checkpointing true：激活重算，显存换时间；长文本/多模态基本必开
--quant_bits 4 + --quant_method bnb：QLoRA 的权重量化；注意 Router/Vision 等子模块未必量化
--lora_rank / --lora_alpha：更新能力与“更新幅度”核心旋钮，通常看 alpha/r 的量级是否合理
--target_modules all-linear：改的层越多越能学，但也越容易过拟合/不稳定；想稳先从 q/k/v/o 或 attention+mlp 关键层开始

推荐的“起步档”（先把闭环跑稳，再追求极限）
lora_rank=8 或 16
lora_alpha=32（或让 alpha/r 维持在 2~8 的量级）
per_device_train_batch_size=1
gradient_accumulation_steps=4~16（看吞吐与显存）
max_length：先 1024/2048 跑通，再上 3072/4096/8192

为什么这套组合能从“加载就 OOM”走向“能跑稳”

3) max_length + checkpointing 负责压激活值：多模态 + 长文本的显存峰值很多时候不在权重，而在激活与临时 buffer

## 根本原因分析： 报错 No available shared memory broadcast block found 意思是：vLLM 的主进程（Process 0）在等待其他 3 张显卡（因为开了 --tensor-parallel-size 4）同步数据，但等了 60 秒都没等到。
造成这种情况通常有两个原因：
1.Docker 共享内存隔离问题：虽然设置了 --shm-size 16g，但在多卡通信（NCCL）高负载下，Docker 的内存隔离机制有时会导致进程间死锁。
2.初始化超时：加载 30B 的模型 + LoRA + 编译 CUDA 图（CUDA Graphs）可能非常慢，超过了默认的 60 秒，导致主进程以为子进程挂了。

记录：
vLLM 启动卡在 KV Cache / CUDA Graph 阶段，60 秒后报 shm_broadcast 超时，TP=4 时，rank0 在等其余 GPU 同步共享内存，但始终未完成

原因总结
Docker 的 IPC / SHM 隔离在高负载 NCCL 下不稳定
CUDA Graph 编译 + LoRA + 30B 初始化时间 超过 vLLM 默认容忍时间

解决方向:增大 --shm-size 或使用 --ipc=host,接受“启动慢但运行稳”的性能-鲁棒性权衡.

第三方复现时的排查顺序（从“最省时间”到“更激进”）
1) 先确认是不是“慢”而不是“挂”：日志里出现 torch.compile/graph capturing 进度条，说明在初始化重活
2) 容器优先改 IPC：--ipc=host 往往比继续加 --shm-size 更有效
3) 先关掉编译/图捕获做排障：加 --enforce-eager（牺牲延迟，换快速验证与稳定启动）
4) 降低启动期显存压力：临时把 --gpu-memory-utilization 下调、把 --max-model-len 下调

通过对比结果得出的两个结论：
一：为什么vllm启动慢？


1.搬运巨石 (Model Weights Loading)
耗时原因：硬盘 I/O 和 PCIe 带宽。
过程：Qwen3-VL-30B 的权重文件大概有 60GB+。
它必须先从硬盘读到 CPU 内存。
然后通过 PCIe 总线搬运到 4 张显卡的显存里。
因为开了 Tensor Parallelism=4，系统还得把每一层的权重切成 4 份，精准地分发给对应的显卡。类比:一头大象切成四块分别装进四个冰箱，动作必须很细致。

2. 团队握手 (NCCL/Distributed Init)
耗时原因：多卡通信同步。
过程：4 张显卡必须组成一个通信环（Ring）。
它们需要通过 NCCL 协议建立连接，确认彼此都在线。
之前遇到的 shm_broadcast 报错就是死在这个阶段。现在虽然通了，但它们需要进行多次“握手”来校准时钟和通信缓冲区，确保推理时数据同步误差在微秒级。

3. 圈地运动 (KV Cache Profiling) —— vLLM 特有的慢
耗时原因：为了极致的推理速度，先做复杂的数学计算。
过程：这是 vLLM 最核心的黑科技（PagedAttention）。
启动时，vLLM 会先跑一次“模拟推理”，测量模型到底占多少显存。
然后它会把剩余所有的显存（根据你的 0.78 参数）划分成成千上万个小块（Pages/Blocks）。
它在计算到底能同时容纳多少个 token？”。这个过程就像是预先给几千个抽屉贴标签，非常消耗 CPU 计算资源。

4. 编译图纸 (CUDA Graph Capture) —— 最慢的一环
耗时原因：JIT 编译。
过程：为了让后续推理快如闪电，vLLM 不会每次都解释执行 Python 代码。
它会在启动时，针对不同的输入长度（Batch Size 1, 2, 4... Sequence Length 128, 256...）录制并编译好 CUDA 执行图。
这就像是把路书背下来，而不是边走边看地图。这个“背书”的过程非常慢，但能让的推理延迟（Latency）降低 30% 以上。


2.这是一个非常的 性能-鲁棒性权衡（Performance-Robustness Trade-off） 现象。 统称(Trade-off)

从的 nvtop 来看，显卡负载均衡（TP=4 工作正常），vLLM 服务响应全是 200 OK，说明基础设施层没有问题。问题出在模型层的“性格变化”上。

微调后的模型变得“描述更全面”，说明它学到了你想要的数据分布（Chain-of-Thought 思维链或详细描述风格），但“偶尔识别不出”说明它出现了过拟合（Overfitting）或者灾难性遗忘（Catastrophic Forgetting）**的早期症状。

以下是导致这种情况的 3 个核心原因及不重新训练的解决办法：

原因一：LoRA 的“强迫症” (过度拟合描述模式)
你现在的 LoRA 权重可能让模型觉得：“如果我不能写出一大段详细的描述，那我就不知道该说什么了。” 当遇到一些特征不明显、或者与训练集差异较大的图片时，底座模型原本可能会说“这是一张模糊的图片”，但挂载 LoRA 后，模型被强制要求“详细描述”，两者冲突导致模型直接崩了（输出为空，或直接输出结束符 <|im_end|>）。

 解决方法：调整 LoRA 的“混合比例” (LoRA Scaling)
vLLM 支持在推理时动态调整 LoRA 的生效程度。你可以把 LoRA 当作一个“调料”，现在 1.0 的浓度太咸了，我们降到 0.7 或 0.8 试试。这通常能找回底座模型的通用能力（稳定性），同时保留微调后的描述能力。

修改你的推理请求（API调用）：

在发送请求的 JSON body 中，尝试添加或修改 scale 参数（如果 vLLM 版本支持），或者在 Prompt 中引导。但最直接的是在启动 Docker 时（或者如果不重启，就在代码层处理）。

如果使用 OpenAI SDK (Python)，vLLM 实际上很难直接传 lora_scale。建议采用以下策略：

降低 Temperature (最重要)： 现在的日志显示你用的是默认 temperature: 0.7。对于警务、OCR、信息提取类任务，0.7 太高了。高温度会让模型在“识别不出”时尝试“瞎猜”，导致概率分布崩塌，最后什么都出不来。

建议：将 temperature 设为 0.1 或 0.01。 我在业务中设置的是0.

原因：强迫模型选择概率最高的 token，极大提高稳定性。

Python

# 修改前端或后端的调用参数
generation_params = {
    "temperature": 0.1,  # 降温，求稳
    "top_p": 0.95,
    "max_tokens": 1024
}
原因二：视觉 Token 的“挤压” (Context Limit)
Qwen2-VL/Qwen3-VL 处理图片的方式是动态分辨率。

如果用户上传了一张长图（例如警务卷宗、长网页截图），模型会把它切成几十个 patch。

一张高清图可能会消耗 3000-5000 个 Token。

你之前的启动命令里 --max-model-len 设为了 8192（为了防显存溢出）。

故障场景： 如果 图片 Token (4000) + Prompt (1000) + 预期输出 (4000) > 8192，vLLM 会强制截断，或者模型发现显存 KV Cache 不够，可能会表现为“识别不出”或直接报错。

 解决方法：检查图片分辨率或提升限制
前端限制：在传给模型前，确保图片长边不要太夸张（比如限制在 2048px 以内），这能显著减少 token 消耗。

观察日志：当你发现“识别不出”的时候，去 vLLM 的日志里看有没有 Input prompt is too long 的警告。

原因三：缺少“负样本” (训练数据偏差)
你微调的数据集（huzhou_police...）可能全是高质量、信息丰富的图片。模型学到了：“图片里一定有东西”。 当推理时来了一张模糊的、或者空白的、或者没啥信息的图，模型在它的“知识库”里找不到对应这种“无聊图片”的模式，于是它困惑了，选择了沉默。

 解决方法：System Prompt 兜底
既然不重训，就用 Prompt 来补救。在 System Prompt 里明确告诉它遇到坏图该怎么办。

修改你的 Prompt：

server {
    listen 9000;
    server_name _;

    root /data;

    client_max_body_size 50m;

    proxy_connect_timeout 30s;
    proxy_send_timeout 600s;
    proxy_read_timeout 600s;
    send_timeout 600s;

    proxy_http_version 1.1;
    proxy_set_header Connection "";
    proxy_buffering off;

    # ========= 1) /report/batches：加密码 =========

    # 统一入口：没带 / 的请求跳转到带 /，避免 alias/try_files 边界问题
    location = /report/batches {
        return 301 /report/batches/;
    }

    # 真实内容：加 Basic Auth
    location ^~ /report/batches/ {
        alias /data/logs/batches/;
        index index.html;
        try_files $uri $uri/ =404;

        auth_basic "Restricted";
        auth_basic_user_file /etc/nginx/.htpasswd_report;

        # 可选：叠加 IP 白名单（示例：公司出口 IP、或办公网段）
        # satisfy all;             # 同时满足 auth + allow 才放行（默认就是 all）
        # allow 1.2.3.4;
        # allow 5.6.7.0/24;
        # deny all;
    }

    # ========= 图片=========
    location /images/ {
        alias /data/infer_images/;
        autoindex off;
        access_log off;
        expires 1d;

        auth_basic "Restricted";
        auth_basic_user_file /etc/nginx/.htpasswd_report;
    }

    # ========= 3) 日志：建议也保护（否则 /logs/ 可能暴露内部信息）=========
    location /logs/ {
        alias /data/logs/;
        autoindex on;
        default_type text/plain;

        auth_basic "Restricted";
        auth_basic_user_file /etc/nginx/.htpasswd_report;
    }

    # ========= 4) API：不加密码（按你要求）=========
    location /api/vllm/ {
        proxy_pass http://127.0.0.1:8000/;

        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;

        proxy_http_version 1.1;
        proxy_set_header Connection "";
        proxy_buffering off;
    }

    location /api/train/ {
        proxy_pass http://127.0.0.1:8100/;
    }

    # ========= 5) 防止脚本被当静态下载（你原来的规则保留）=========
    location ~* \.(py|sh)$ {
        default_type text/plain;
    }
}   


训练平台网关配置 (Nginx)概述：Nginx 作为训练平台的独立网关运行，监听 9000 端口。它负责将外部请求路由至内部的推理服务 (vLLM) 和训练后端，同时对敏感的日志和数据集图片目录实施了访问控制。核心策略：端口监听：9000 (独立服务端口)长连接支持：配置了 600s (10分钟) 的超长读写超时时间，适配 AI 模型推理和训练任务的长耗时特性。安全防护：对日志 (/logs/) 和图片 (/images/) 目录启用了 HTTP Basic Auth 密码认证，防止敏感数据未授权访问。路由规则详情：URL 路径映射/转发目标权限控制说明/api/vllm/http://127.0.0.1:8000/公开推理服务：转发至 vLLM 接口 (OpenAI 兼容)/api/train/http://127.0.0.1:8100/公开训练后端：转发至训练控制接口/report/batches//data/logs/batches/加密 (Basic Auth)训练过程中的批次日志报告/images//data/infer_images/加密 (Basic Auth)推理生成的图片结果 (缓存1天)/logs//data/logs/加密 (Basic Auth)系统完整日志目录 (开启目录索引)关键配置代码 (/etc/nginx/conf.d/train_platform.conf)：Nginxserver {
    listen 9000;

# 1. vLLM 推理接口转发

    proxy_pass http://127.0.0.1:8000/;

}


缓存与中间件配置 (Redis)
概述： 采用 Docker 容器化部署 Redis 7.0，作为系统的高速缓存层。该服务仅面向宿主机内部开放，确保了数据安全性。

基础信息：

部署方式：Docker Container

镜像版本：redis:7

容器名称：redis (Container ID: 9aeb1fcb770c)

端口映射：127.0.0.1:6379:6379 (安全强化：仅允许本地访问，禁止公网直连)

关键策略：

网络隔离：通过绑定 127.0.0.1，确保只有本机服务（如 Java 后端、Nginx）可以连接 Redis，有效防止了未授权的外部访问攻击。

数据持久化：(根据默认配置) 采用默认 RDB 策略，在指定时间间隔内保存快照。

[S5] vLLM kernel/graph 探索（如何把“现象”变成“可解释”）
目标：解释 3 件事——为什么 Decode 慢、哪些 kernel 是瓶颈、torch.compile/CUDA Graph 到底省了什么。

torch.compile vs CUDA Graph（结合你日志里的两段关键行）
1) torch.compile：把模型的一段计算图编译成更少的 kernel 调度与更高效的融合形式（日志里会出现 “Compiling a graph for compile range ... takes ... s”）
2) CUDA Graph capture：把一段固定形状的 GPU 调用序列“录制”下来，后续复用以减少 kernel launch 开销（日志里会出现 “Capturing CUDA graphs ...”）
3) 为什么会触发 shm_broadcast 超时：多卡 TP=4 时，各 rank 在做 compile/capture 的耗时可能不一致；rank0 等其他 rank 同步共享内存块时，默认 60s 容忍度会被打穿

MoE 提示的含义（和 kernel 选择有关）
WARNING ... fused_moe.py ... Using default MoE config. Performance might be sub-optimal! Config file not found at ... device_name=NVIDIA_L20.json
这代表 fused MoE kernel 的“设备特定调参缓存”没命中，回退到默认参数组合；通常不影响正确性，但会影响吞吐上限与尾延迟。做极限优化时要把它当作一个“性能告警”而不是“功能错误”。

怎么理解 “By default this is no longer used”
1) 这不等于 PagedAttention 思想没用：PagedAttention 更像是“KV Cache 的页式管理策略”，而不是某一个固定 kernel 名字
2) vLLM 的实现会随版本/后端切换（例如不同 attention backend、不同 kernel fusion 策略），所以这个脚本更适合做“单 kernel 对比”而不是“线上吞吐预测”
3) 做容量规划要看 serving/benchmark_serving 的 tok/s；做性能定位才看 kernel benchmark + profile

Decode 为什么往往是瓶颈（结合上面的 Output tok/s）
1) Decode 每步只生成 1 token，但要反复读 KV Cache（多头、多层、长上下文时读带宽压力巨大）
2) Prefill 更像“大矩阵乘 + 更容易吃满算力”；Decode 更像“带宽受限 + 小批量 kernel 启动频繁”
3) 多模态模型在 Prefill 还可能包含 Vision Encoder，导致启动期的 compile/graph capturing 更重

SM89 = NVIDIA L20 / L40 / RTX 4090 (Ada Lovelace 架构)

2. 什么时候该用 FP8？
仔细看表格的前几行（低负载）：
Batch Size 1~64: torch-bf16 (73 T) > fp8 (46 T)
Batch Size 128: torch-bf16 (85 T) >> fp8 (60 T)
Batch Size 256: 拐点出现！ FP8 (115 T) 反超 BF16 (88 T) 

洞察：
FP8 不是万能的：在小并发（Batch < 256）时，FP8 因为有量化/反量化的额外开销 (Overhead)，速度反而不如直接跑 BF16。
FP8 是为吞吐而生：只有当并发请求堆积到 256 以上 时，Tensor Core 的 FP8 引擎才能抵消掉量化的开销，开始展现“暴力美学”。
3. 给 Infra 工程师的最终建议
基于这几天我们跑的所有 benchmark（吞吐量、带宽、缓存、算力），我对你这台 4x L20 + Qwen3-VL-30B 的机器给出最终的部署策略建议：
场景 A：内部小范围测试 / 个人助理
策略：保持 BF16 (默认)。
理由：并发低，BF16 延迟更低，且不需要折腾量化，精度最高。
场景 B：公司级公共 API / RAG 知识库
策略：必须上 FP8 且开启 Prefix Caching。

理由：
FP8 保证了高并发下的吞吐算力（179 TFLOPS）。
Prefix Caching 解决了长文档的 I/O 瓶颈（11,000 token/s）。
这套能把 L20 的性能榨干到极致。

结论：Qwen-30B 模型大概有 60~70 层。
计算：60层 × (60μs + 55μs) ≈ 6900μs = 6.9 毫秒。
这意味着即使我给每一层都挂上 LoRA，生成每个 token 的额外延迟也只有不到7毫秒。对于用户来说，这完全是无感的。

惊人的平稳性:
看数据，无论 Hidden Size 是 1792、5120 还是巨大的 13824，延迟竟然都死死卡在 60μs 左右。
说明在L20 这种强力显卡面前，LoRA 的计算量太小了，显卡甚至还没“热身”就跑完了。现在的瓶颈其实是 Python 发送指令的通讯耗时（Overhead），而不是计算耗时。
真正部署上线（使用 C++ Runner 和 CUDA Graph）时，这个 60μs 还会大幅下降。


至此，我对这台 4*L20 64核 服务器 的基本实验都已完成。infra架构文档里有以下结论：

 基础能力：Qwen-30B 吞吐量达 865 tokens/s，足以支撑实时对话。

 长文能力：显存带宽充裕 (371GB/s)，配合 vLLM Prefix Caching，RAG 检索速度提升 4.2 倍。

 上限： FP8 算力高达 202 TFLOPS，未来量化后性能可翻倍。

  2.Chunked-Prefills (分块预填充)
应用场景：混合了极长 Prompt（如 RAG 检索增强生成）和短对话的场景，防止长请求阻塞短请求。
实验操作：
调整参数：--max-num-batched-tokens (也称为 Chunk Size)。
测试场景：构造一个包含长 Prompt (prefill) 和正在进行的生成任务 (decode) 的混合负载。
调优：
设置较小的值（如 512）：优先保证 decode 任务的流畅性，降低 ITL (Inter-Token Latency)。
设置较大的值（如 4096+）：优先保证 prefill 的吞吐量，减少 TTFT。
观察指标：使用 benchmark_throughput.py 观察在不同参数下的 P99 ITL 延迟数据。

[S6] 附录：参数速查 + 外部资料

LoRA/QLoRA 参数速查（最常被问的那几个）
r（rank）：LoRA 低秩维度，越大越能学，显存/训练不稳定风险也越高
alpha：缩放因子，经验上更关心 alpha/r 的量级是否合理
target_modules：哪些线性层打补丁；越“全”越容易学到，但也越容易引入不稳定或过拟合
gradient_checkpointing：激活重算换显存；长上下文/多模态优先打开
max_length：把它当成“显存压力阀”；先跑通小值，再逐步抬高

vLLM 参数速查（与稳定性/吞吐最相关）
--tensor-parallel-size：跨卡切分权重，决定多卡同步与吞吐上限
--gpu-memory-utilization：KV Cache 与图捕获的预留；偏高更容易 OOM/抖动
--max-model-len：上限越大，KV Cache 预留越多；容量规划要以它为基准
--enable-lora + --lora-modules：不合并权重，直接挂载 LoRA 做回归对照
--ipc=host / --shm-size：多卡启动不稳时优先排查（见 [S4]）
